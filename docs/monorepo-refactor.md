# 모노레포 전환 기록

## 문서 목적

이 문서는 `Ocean`, `Ocean Admin`, `Piuda`로 분리되어 운영 중인 프론트엔드 프로젝트들을 모노레포 구조로 전환하는 과정을 기록하기 위한 문서다.

단순히 코드 구조를 바꾸는 작업 로그가 아니라, 왜 이 전환이 필요했는지, 어떤 공통 요소를 식별했는지, 무엇을 루트 공통 영역에서 관리할지, 리팩토링 과정에서 어떤 의사결정을 했는지를 단계적으로 정리하는 데 목적이 있다.

추후에는 이 문서를 바탕으로 포트폴리오용 리팩토링 사례를 정리할 예정이다.

## 배경 상황

현재 서비스는 성격이 다른 3개의 프론트엔드 프로젝트로 구성되어 있다.

1. `Ocean`
   - 현장에서 사용하는 PWA 앱
   - 서비스 데이터를 등록하거나 업로드하는 역할을 담당한다.
2. `Ocean Admin`
   - 백오피스 프로젝트
   - `Ocean`에서 올라온 데이터를 검수, 수정, 관리하는 역할을 담당한다.
3. `Piuda`
   - 대시보드 프로젝트
   - `Ocean Admin`에서 관리하는 데이터를 기준으로 시각화와 모니터링 기능을 제공한다.

이 3개 프로젝트는 역할은 다르지만 하나의 서비스 도메인 안에서 연결되어 있다. 실제 데이터 흐름도 `Ocean`에서 생성된 데이터가 `Ocean Admin`에서 관리되고, 다시 `Piuda`에서 시각화되는 구조로 이어진다.

또한 프론트엔드 저장소는 각각 분리되어 있지만, 백엔드는 하나의 레포지토리에서 세 프로젝트의 요청을 함께 처리하고 있다. 즉, 서비스 관점에서는 하나의 도메인으로 움직이고 있지만 프론트엔드만 물리적으로 분리된 상태라고 볼 수 있다.

## 현재 구조의 특징

- 3개의 프로젝트가 서로 다른 리포지토리에서 관리되고 있다.
- 도메인 모델은 강하게 연결되어 있지만 코드베이스는 분리되어 있다.
- 백엔드는 이미 하나의 레포지토리에서 통합적으로 운영되고 있다.
- 프론트엔드에서는 프로젝트별로 개발, 배포, 공통 로직 관리가 각각 따로 이뤄지고 있다.

이 구조는 프로젝트 초기에 역할을 빠르게 분리하는 데는 유리할 수 있지만, 서비스가 확장되고 프로젝트 간 연결이 많아질수록 관리 비용이 점점 커지는 문제가 있다.

## 필요성

대시보드 프로젝트와 PWA 앱, 그리고 대시보드와 PWA 앱을 관리하는 백오피스 프로젝트까지 총 3개의 프론트엔드 프로젝트가 존재하지만, 현재는 각각 다른 리포지토리로 관리되고 있다. 이로 인해 서비스는 하나의 도메인으로 동작함에도 불구하고 개발 경험과 운영 구조는 분절되어 있다.

이 분리는 다음과 같은 문제를 만든다.

1. 공통 도메인에 대한 관리 포인트가 흩어진다.
   - 세 프로젝트가 같은 데이터를 서로 다른 관점에서 다루는데, 리포지토리가 분리되어 있으면 타입, 상수, API 스펙 해석, 상태값 정의가 각 저장소에 중복되기 쉽다.
   - 그 결과 같은 도메인을 다루면서도 프로젝트마다 미세하게 다른 구현이 생길 가능성이 높아진다.
2. 변경 전파 비용이 크다.
   - 데이터 구조나 API 응답 형식이 바뀌면 `Ocean`, `Ocean Admin`, `Piuda`가 함께 영향을 받는다.
   - 하지만 저장소가 분리되어 있으면 변경 내용을 각각의 레포지토리에서 따로 반영해야 하므로 수정 누락, 반영 시차, 커뮤니케이션 비용이 발생한다.
3. 공통 로직과 자산이 중복된다.
   - 인증 처리, API 클라이언트, 디자인 토큰, 공통 UI 컴포넌트, 유틸 함수, 환경 설정 규칙처럼 여러 프로젝트에서 반복되는 요소가 생긴다.
   - 현재 구조에서는 이런 공통 요소를 중앙에서 관리하기 어렵기 때문에 복사 또는 수동 동기화 방식으로 유지될 가능성이 높다.
4. 프로젝트 간 정합성을 보장하기 어렵다.
   - 한 기능이 실제 서비스에서 어떻게 이어지는지 보려면 여러 저장소를 오가며 확인해야 한다.
   - 특히 `업로드 -> 관리 -> 시각화`처럼 하나의 사용자 흐름이 여러 프로젝트를 관통하는 경우, 전체 흐름을 한 번에 검증하기 어렵다.
5. 개발 생산성과 유지보수성이 떨어진다.
   - 저장소마다 의존성, 스크립트, 설정 방식이 다르면 온보딩 비용이 커지고 반복 작업도 많아진다.
   - 공통 개선 사항이 있어도 각 프로젝트에 개별 적용해야 하므로 구조 개선의 효율이 낮아진다.
6. 아키텍처 관점에서 프론트엔드만 불필요하게 분리되어 있다.
   - 백엔드는 이미 하나의 레포지토리에서 세 프로젝트의 요청을 함께 처리하고 있다.
   - 반면 프론트엔드는 동일 도메인을 다루면서도 저장소가 나뉘어 있어, 실제 서비스 구조와 개발 구조 사이에 괴리가 생긴 상태다.

따라서 이번 모노레포 전환은 단순한 저장소 통합이 아니라, 같은 도메인을 공유하는 프론트엔드 애플리케이션들을 하나의 구조 안에서 일관되게 관리하기 위한 기반을 만드는 작업이라고 볼 수 있다.

특히 이번 리팩토링에서는 다음 질문에 답하는 방향으로 구조를 정리할 예정이다.

- 세 프로젝트가 진짜로 공유하는 공통 요소는 무엇인가
- 어떤 코드는 앱 내부에 남기고, 어떤 코드는 루트 공통 패키지로 올려야 하는가
- 도메인 모델, API 타입, 디자인 시스템, 유틸리티를 어떤 단위로 분리해야 재사용성과 독립성을 모두 확보할 수 있는가
- 프로젝트 간 결합도를 높이지 않으면서도 관리 비용을 줄이려면 어떤 폴더 구조와 의존성 규칙이 필요한가

이 문서는 위 질문들에 대한 판단 근거와 실제 리팩토링 과정을 순차적으로 기록하는 기준 문서로 사용한다.
